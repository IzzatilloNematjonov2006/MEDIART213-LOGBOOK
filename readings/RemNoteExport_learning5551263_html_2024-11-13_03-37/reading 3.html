  • Reading summary and notes by AI:
  • Brad Frost reflects on a year spent integrating accessibility into web design, discussing the principles and practical methods that guided his work. His primary approach, progressive enhancement, emphasizes building web content in layers, starting with a robust HTML foundation and adding CSS and JavaScript for functionality. This approach ensures that content remains accessible, even if some layers fail. Through various examples, Frost explains how using semantic HTML improves screen reader compatibility and enhances the experience for users relying on assistive technologies. His journey highlights the need for empathy-driven, user-centered design and a commitment to inclusive practices.
  • Important Notes
    • Progressive Enhancement: Build web interfaces by layering functionality, beginning with a basic HTML structure. CSS and JavaScript should be added on top, allowing the content to remain accessible even if JavaScript is disabled or fails​.
    • Semantic HTML: Meaningful use of HTML elements, such as headings and buttons, improves accessibility. Frost emphasizes structuring HTML based on content purpose rather than appearance, aiding screen readers and providing a logical navigation structure.
    • User-Centered Accessibility: Designing with empathy by considering a broad range of user needs fosters better accessibility. Involving users early in the design process and testing with diverse audiences ensures that interfaces are more inclusive and responsive to real needs​
  • Notes by me:
  • HTML is semantic, or in other words, it is descriptive and provides meaning — each HTML element describes the type of content it presents.
  • By using correct elements, your document content will have conveyable structure and meaning.
  • f you don’t use proper landmarks (exposed to screen readers via semantic HTML elements like <nav> and <main> and <header>), screen reader users may not be able to efficiently navigate the page and would have to search for their areas of interest more tediously.
  • Headings also provide an outline or “skeleton” of the page that users can navigate using hot keys.
  • But if you don’t use a <button> to create a button, and you choose to use a <div>, for example, instead, you lose all the built-in semantics and keyboard interactivity, making the button completely inaccessible to screen readers.
  • ARIA attributes are possibly the most powerful tool in our accessibility arsenal.
  • If you can use a native HTML element or attribute with the semantics and behavior you require already built in, instead of re-purposing an element and adding an ARIA role, state or property to make it accessible, then do so.
  • — semantic HTML! — and then layering styles (CSS) and functionality (JavaScript!)
  • Use HTML. Enhance with CSS. Leverage the power of JavaScript. And include your users in your design process as early as possible. And always look at your work through the lens of inclusion.
